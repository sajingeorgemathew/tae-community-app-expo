### 1.7 Key RLS Policies Summary

#### profiles
- **Not in repo migrations.** Two policies found in migrations:
  - `"Admins can update is_disabled"` — allows admin UPDATE on all profile columns (migration `20250130`)
  - `"Admins can update any profile"` — allows admin UPDATE on all profile columns (migration `20250211`, replaces/overlaps the above)
- **Critical gap:** No SELECT, INSERT, or user-self-UPDATE policies are in migrations. These must be verified in the Supabase dashboard.

#### posts
- SELECT: all authenticated
- INSERT: `author_id = auth.uid()`
- UPDATE: `author_id = auth.uid()`
- DELETE: author OR admin

#### conversations / messages
- SELECT: only conversation members (via `conversation_members` join)
- INSERT messages: sender must be member, `sender_id = auth.uid()`
- DELETE messages: sender only (no admin override)

#### Storage: message-media
- SELECT: conversation members only (path-based lookup)
- INSERT: conversation members only (path-based)
- DELETE: message sender only

#### Storage: profile-avatars
- SELECT: any authenticated user (needed for viewing avatars)
- INSERT/UPDATE/DELETE: path owner only (via `is_avatar_path_owner()`)

#### Storage: post-media
- DELETE: uploader OR admin (migration `20250127`)
- **INSERT/SELECT policies not in repo migrations — must verify in dashboard**

## 2. Findings Table

### F-01: Admin Route Has No Server-Side Protection

| | |
|---|---|
| **Severity** | **Critical** |
| **Evidence** | `src/middleware.ts:21-38` — middleware only checks authentication and `is_disabled`, never checks `role`. `src/app/app/admin/page.tsx:136-142` — admin check is a client-side `getSession()` + profile query. |
| **Impact** | The `/app/admin` route is accessible to any authenticated user. The admin gate is purely cosmetic (client-side state `isAdmin`). While individual operations (like updating roles) are protected by RLS policies that check `role = 'admin'`, any authenticated user can navigate to `/app/admin` and see the admin UI skeleton load. The true protection relies entirely on RLS, which is correctly configured for write operations but means the admin page HTML/JS bundle is served to all users. |
| **Repro idea** | 1. Sign in as a normal member. 2. Navigate to `/app/admin`. 3. Page loads, client code fetches posts/users — the fetches themselves succeed (SELECT policies allow all authenticated users to read profiles and posts). 4. The client-side `isAdmin` check hides the UI, but a user modifying the client JS or using browser devtools can see the fetched data. |
| **Fix** | Add a server-side role check in `middleware.ts` for `/app/admin`. Query `profiles.role` and redirect non-admins before the page renders. This does NOT require `service_role` key — the existing middleware Supabase client (using the user's session) already queries profiles for `is_disabled`. Add `role` to that same query. |

### F-02: Profiles Table — Missing Base RLS Policies in Migrations

| | |
|---|---|
| **Severity** | **Critical** |
| **Evidence** | No migration creates the `profiles` table or its base SELECT/INSERT/UPDATE policies. The table is referenced across all 22 migrations. Only admin-specific UPDATE policies exist in migrations (`20250130`, `20250211`). |
| **Impact** | If profiles RLS is misconfigured in the dashboard (or was never properly set up), any authenticated user could potentially UPDATE any profile's `role` field to `'admin'`, achieving full privilege escalation. The CHECK constraint `profiles_role_check` (migration `20250211`) limits values to `member/tutor/admin/alumni` but doesn't prevent a user from setting their own role to `admin`. |
| **Repro idea** | 1. From browser console: `supabase.from('profiles').update({ role: 'admin' }).eq('id', '<own-user-id>')`. 2. If no user-self-UPDATE policy restricts which columns can be modified, this succeeds. 3. User now has admin privileges across the entire app. |
| **Fix** | **Must verify in Supabase dashboard immediately.** Ensure: (a) A user-self-UPDATE policy exists that restricts users to updating only safe columns (`full_name`, `program`, `grad_year`, `avatar_path`, `headline`, `skills`) and explicitly excludes `role`, `is_disabled`, `is_listed_as_tutor`. (b) Export these policies into a migration file for version control. This does NOT require `service_role` — it's a dashboard RLS policy change. |

### F-03: Duplicate/Overlapping Admin UPDATE Policies on Profiles

| | |
|---|---|
| **Severity** | **High** |
| **Evidence** | Migration `20250130` creates `"Admins can update is_disabled"` with broad USING/WITH CHECK that allows admin to update ANY column. Migration `20250211` creates `"Admins can update any profile"` with the same logic. Both policies exist simultaneously. |
| **Impact** | Both policies allow admins to update ALL columns, not just the intended ones. An admin could change another admin's role to `'member'`, effectively locking them out. The `20250130` policy was named for `is_disabled` but actually grants blanket UPDATE. Additionally, the overlap may cause confusion during policy audits. |
| **Fix** | Drop the older `"Admins can update is_disabled"` policy. Optionally scope the remaining admin UPDATE policy to specific columns using a WITH CHECK that validates only intended columns changed. This is a dashboard/migration change only. |

### F-04: User Can Update Own Profile Role via Direct Supabase Call

| | |
|---|---|
| **Severity** | **Critical** (depends on dashboard state — see F-02) |
| **Evidence** | `src/app/app/me/page.tsx:337-343` — the `handleSave` function sends `supabase.from('profiles').update(updates).eq('id', profile.id)` where `updates` includes `full_name`, `program`, `grad_year`, `headline`, `skills`, `avatar_path`. ### F-05: `getSession()` Used Instead of `getUser()` for Auth Checks in Client Components

| | |
|---|---|
| **Severity** | **Medium** |
| **Evidence** | `src/app/app/admin/page.tsx:126`, `src/app/app/messages/page.tsx:517`, `src/app/app/me/page.tsx:107`, `src/app/app/directory/page.tsx:79`, `src/app/app/feed/page.tsx:96`, `src/app/app/feed/new/page.tsx:48`, `src/app/app/page.tsx:144`, `src/app/app/profile/[id]/page.tsx:80` — all use `supabase.auth.getSession()`. |
| **Impact** | `getSession()` reads from local storage and does NOT verify the JWT with Supabase servers. A user could have a revoked/expired session that still appears valid locally. The middleware correctly uses `getUser()` (line 16), but client components bypass this by using `getSession()`. This means a disabled user who was already on the page before being disabled could continue to interact until they navigate away and trigger middleware. |
| **Repro idea** | 1. Admin disables a user. 2. The disabled user already has the `/app/messages` page open. 3. User continues sending messages because the client never re-validates the session with the server. Only on next navigation will middleware catch it. |
| **Fix** | Replace `getSession()` with `getUser()` in security-sensitive client operations (especially admin page and profile updates). For read-only pages where performance matters, `getSession()` is acceptable. Note: the anon key + RLS still provides server-side protection, so this is defense-in-depth. |

### F-07: Conversation Member Can Add Themselves to Any Conversation

| | |
|---|---|
| **Severity** | **Medium** |
| **Evidence** | `20250203_create_messaging_tables.sql` — `conversation_members` INSERT policy: `with check (user_id = auth.uid())`. The `conversations` INSERT policy: `with check (true)`. |
| **Impact** | Any authenticated user can: (a) Create a new conversation. (b) Insert themselves as a member of ANY existing conversation (the INSERT policy only checks `user_id = auth.uid()`, not whether the conversation was created for them). Once a member, they can read all messages in that conversation. |
| **Repro idea** | 1. User A obtains a conversation ID (e.g., from a URL shared by another user, or by brute-forcing UUIDs). 2. User A runs: `supabase.from('conversation_members').insert({ conversation_id: '<target-conv-id>', user_id: '<own-id>' })`. 3. User A can now read all messages in that conversation. |
| **Fix** | Restrict `conversation_members` INSERT to only allow insertion via the `create_conversation_1to1` RPC (which is SECURITY DEFINER and already handles this properly). Options: (a) Remove the direct INSERT policy on `conversation_members` entirely — the RPC bypasses RLS. (b) Or add a condition that prevents insertion into conversations that already have 2 members. This is a migration-only fix. |

### F-08: No Rate Limiting on Any Client Operation

| | |
|---|---|
| **Severity** | **Medium** |
| **Evidence** | All operations go through the browser Supabase client (`src/lib/supabaseClient.ts`). No rate limiting exists at the application level. No Next.js API routes exist that could enforce rate limits. |
| **Impact** | A malicious user could: (a) Spam posts/comments/reactions/messages at unlimited rate. (b) Mass-create conversations with all users. (c) Flood the presence table with heartbeat upserts. (d) Exhaust Supabase API quota. |
| **Fix** | Implement rate limiting via Supabase Edge Functions or introduce Next.js API routes with rate limiting middleware. For immediate mitigation, enable Supabase's built-in rate limiting in the dashboard (Project Settings > API > Rate Limiting). |

### F-09: Polling Loops Create Scaling Bottleneck

| | |
|---|---|
| **Severity** | **Medium** |
| **Evidence** | `src/app/app/messages/page.tsx:16-17` — `CONVERSATION_POLL_INTERVAL = 6000` (6s), `MESSAGES_POLL_INTERVAL = 3000` (3s). `src/lib/AppMetricsContext.tsx:6` — `POLL_INTERVAL_MS = 8000` (8s). `src/lib/usePresenceHeartbeat.ts:6` — `HEARTBEAT_INTERVAL_MS = 45000` (45s). |
| **Impact** | Per active user with messages open: 1 request/3s (messages) + 1 request/6s (conversations) + 1 request/6s (tick state) + 1 request/8s (app metrics) + 1 request/45s (presence) = ~37 requests/minute. With 100 concurrent users, this is ~3,700 req/min to Supabase. The `get_my_conversations()` RPC is particularly expensive — it joins 4 tables with correlated subqueries for unread counts. |
| **Fix** | (a) Replace polling with Supabase Realtime subscriptions for messages and conversations. (b) Increase poll intervals (e.g., conversations to 15s, metrics to 30s). (c) Add visibility-aware polling (already partially done — `isVisibleRef` check exists). (d) For the `get_my_conversations()` RPC, consider caching or materialized views for unread counts. |

| **Evidence** | `20250203_create_messaging_tables.sql` — `conversation_members` SELECT: `using (user_id = auth.uid())`. |
| **Impact** | Users can only see their own membership rows, not who else is in the conversation. This is by design for 1:1 chats (the `get_my_conversations()` RPC handles this via SECURITY DEFINER). However, if group chats are ever added, this policy would need revision. No current vulnerability. |
| **Fix** | No action required for 1:1 architecture. Document this constraint for future group chat implementation. |

### F-20: Admin Can Delete Any Post But Not Edit It

| | |
|---|---|
| **Severity** | **Low** |
| **Evidence** | `20250125_create_posts_table.sql` — UPDATE: `author_id = auth.uid()` (no admin bypass). DELETE: author OR admin. |
| **Impact** | Admin moderation is deletion-only — no ability to edit offensive content while preserving the post. This is a UX limitation, not a security issue. Current behavior is actually safer (prevents admin from impersonating users by editing their posts). |
| **Fix** | No action required from a security perspective. If edit capability is desired, add an admin UPDATE policy with an audit trail. |

### F-21: `create_conversation_1to1` Allows Conversation with Disabled Users

| | |
|---|---|
| **Severity** | **Low** |
| **Evidence** | `20250203_ticket25_create_conversation_rpc.sql:29-31` — checks `profiles` existence but not `is_disabled`. |
| **Impact** | A user can start a conversation with a disabled user. The disabled user can't access the app (middleware blocks them), but the conversation row gets created. Minor data consistency issue. |
| **Fix** | Add `AND NOT is_disabled` to the profile existence check in the RPC function. |
